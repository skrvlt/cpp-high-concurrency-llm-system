# 基于C++高并发架构与语言大模型的交互系统的开发

## 摘要

随着大语言模型在智能问答、企业知识服务、在线咨询和辅助决策等场景中的广泛应用，面向真实业务环境的智能交互系统不仅需要具备较强的语义理解与文本生成能力，还需要具备较高的并发处理能力、稳定性和可扩展性。传统以脚本语言为主的智能问答服务在快速开发方面具有明显优势，但在高并发网络连接管理、请求调度和资源控制方面往往存在性能瓶颈。针对上述问题，本文设计并实现了一种基于 C++ 高并发服务架构与语言大模型推理服务相结合的智能交互系统。

系统整体采用前后端分离与双后端微服务协同的设计思路。前端通过 Web 页面完成用户交互；接入层采用基于 Linux 平台的 C++ 高性能服务器，利用 Reactor 模式、Epoll I/O 多路复用、线程池、HTTP 有限状态机解析以及连接超时管理等机制，实现高并发连接接入与请求调度；业务智能层采用 Python FastAPI 框架封装大语言模型接口，负责用户语义请求处理、上下文拼接、结果生成与数据返回；数据层使用 MySQL 完成用户信息、会话记录、系统配置与日志等内容的持久化存储。系统通过 HTTP 接口完成 C++ 接入层与 Python AI 服务之间的跨语言通信，在保持底层高性能的同时兼顾上层智能服务开发效率。

在系统实现基础上，本文从功能完整性、并发性能、稳定性和可维护性等方面展开测试。测试结果表明，该系统能够完成用户登录、发起智能问答、记录会话、后台配置管理等核心业务流程，在一定并发压力下能够保持较好的吞吐能力和响应时间，证明了“底层高性能网络架构 + 上层语言大模型能力”相结合的方案具有较强的工程可行性和应用价值。

本文的研究工作对于高校软件工程类毕业设计具有一定参考意义，也为后续进一步引入知识库检索、流式输出、权限隔离和模型服务优化等功能提供了可扩展基础。

**关键词：** C++；高并发；Reactor；Epoll；FastAPI；大语言模型；智能交互系统

## Abstract

With the rapid development of large language models, intelligent interaction systems are being widely used in enterprise knowledge services, online consultation, educational assistance, and data analysis scenarios. In practical applications, such systems are required not only to provide accurate natural language understanding and response generation, but also to support high concurrency, stable operation, and scalable deployment. Script-based AI services are efficient for rapid development, yet they often encounter bottlenecks in network connection management, request scheduling, and system resource control under high-load conditions. To address these issues, this thesis designs and implements an interactive system based on a high-concurrency C++ architecture combined with a large language model service.

The system adopts a front-end and back-end separated architecture with two cooperative backend services. The front-end provides browser-based interaction. The access layer is implemented with a Linux-based C++ high-performance server, which uses the Reactor pattern, Epoll I/O multiplexing, thread pools, HTTP finite-state-machine parsing, and connection timeout management to support high-concurrency request processing. The intelligent business layer is implemented with Python FastAPI, which encapsulates large language model APIs for semantic request processing, context organization, and answer generation. MySQL is used for persistent storage of user information, dialogue records, system parameters, and logs. Cross-language communication between the C++ service and the Python AI service is achieved through HTTP interfaces, balancing low-level performance with upper-layer development flexibility.

Based on the implemented system, this thesis performs functional testing, concurrency testing, stability testing, and maintainability analysis. The results show that the system can complete core processes such as user login, intelligent question answering, dialogue recording, and background configuration management. Under a certain concurrent load, the system maintains acceptable throughput and response latency, which verifies the engineering feasibility of combining a high-performance network framework with large language model capabilities.

This work provides a practical reference for undergraduate software engineering projects and establishes a foundation for future extensions such as knowledge retrieval enhancement, streaming output, access control, and model service optimization.

**Key words:** C++; high concurrency; Reactor; Epoll; FastAPI; large language model; interactive system

## 第1章 绪论

### 1.1 研究背景

近年来，以 ChatGPT、DeepSeek、通义千问、智谱 GLM 等为代表的大语言模型快速发展，推动自然语言处理技术从传统规则驱动和浅层机器学习阶段迈向以预训练模型为核心的智能化阶段。与传统问答系统相比，大语言模型在上下文理解、复杂语义表达、开放域知识组织以及多轮对话生成方面表现更加突出，因此在智能客服、办公助手、知识库问答、教育辅导和行业咨询等场景中具有广阔应用前景。

但从系统工程角度看，模型能力的提升并不等同于应用系统成熟度的提升。一个真实可用的智能交互平台不仅要“会回答”，还要“接得住、转得快、跑得稳、易维护”。尤其在多用户同时访问场景下，系统会出现连接数增长快、请求处理链路变长、上下游接口阻塞、资源回收不及时、日志难以追踪等问题。如果仍采用单体式脚本服务直接承载浏览器接入、请求解析、模型调用和管理功能，往往会在并发能力、稳定性与扩展性方面暴露明显瓶颈。

高并发接入问题在 Web 应用、即时通信、在线考试和智能服务平台中具有共性。传统做法通常将高性能网络接入与上层业务逻辑耦合在同一服务中，虽然前期开发简单，但在后续进行性能调优、模块扩展和故障定位时会付出更高成本。基于此，采用分层设计思想，将高性能网络接入层与大模型业务层拆分，成为一种兼顾性能与开发效率的可行方案。

本课题以“基于 C++ 高并发架构与语言大模型的交互系统的开发”为对象，拟构建一套可用于浏览器问答场景的双后端协同系统。其中，C++ 负责高并发连接接入、事件监听与请求转发，Python FastAPI 负责模型调用、上下文组织和结果返回，通过 HTTP 接口完成跨语言协作。这种架构既保留了 C++ 在高性能网络场景下的优势，又利用了 Python 在 AI 生态中的成熟能力，具有较强的工程研究价值。

### 1.2 研究目的与意义

本文的研究目的在于面向智能问答与业务辅助交互场景，设计并实现一个基于 C++ 高并发接入架构与语言大模型服务相结合的交互系统，使系统在具备自然语言问答能力的同时，能够满足较高并发访问下的连接管理、请求转发和稳定运行需求。围绕这一总体目标，本文希望解决以下几个关键问题：

1. 如何将高并发接入层与大模型服务层解耦，形成职责清晰、便于维护的双后端协同结构。
2. 如何在不显著增加项目实现难度的前提下，引入 Reactor、Epoll、线程池等典型高性能网络技术，使系统具备一定的并发处理能力。
3. 如何通过统一接口契约和跨环境运行设计，使系统不仅能在作者本机演示，而且具备较清晰的部署路径和实验复现条件。

本研究具有以下几方面意义：

1. 在理论层面，探索高性能网络编程技术与大语言模型服务化部署的结合方式，为“底层高性能接入 + 上层智能能力封装”的系统设计提供分析样本。
2. 在工程层面，验证 C++ 接入层与 Python AI 服务协同设计的可行性，为后续类似系统开发提供模块划分、接口组织和运行验证方面的参考。
3. 在实践层面，形成一套适合本科毕业设计实施的完整方案，包括需求分析、总体设计、系统实现、测试验证和论文撰写方法。
4. 在扩展层面，为后续加入知识库检索、流式输出、负载均衡、权限隔离和模型本地化部署等方向预留接口与架构空间。

对于软件工程类本科毕业设计而言，本课题的意义还体现在“兼顾技术深度与项目可落地性”。一方面，它能够体现高性能网络编程、前后端协同、数据库设计和系统测试等计算机专业核心能力；另一方面，它避免了纯算法研究或纯界面开发带来的单一性，更适合作为软件开发类毕业设计选题。

### 1.3 国内外研究现状

#### 1.3.1 国外研究现状

在高性能服务器领域，国外较早开展了事件驱动网络模型、I/O 多路复用机制和高并发服务端编程研究。Epoll、Kqueue、Reactor、Proactor 等模型在工业界已有广泛应用。Linux 平台下基于事件驱动的服务器框架在高并发连接处理方面具有较高效率，已成为互联网服务端开发的重要基础。相关研究和工程实践表明，在高连接数与高并发访问场景中，事件驱动模型在资源利用率和响应性能方面通常优于简单线程直连模型[3][5][6]。

在人工智能服务部署领域，国外研究较早关注模型服务化、推理加速和云端部署方案。近年来，围绕模型网关、推理接口封装、微服务协同和 AI 基础设施优化的研究逐渐增多。FastAPI、gRPC、异步消息队列以及容器化部署等技术被广泛用于智能服务系统搭建。随着大模型应用从研究验证阶段走向业务落地阶段，如何构建兼具接口稳定性、部署灵活性和多服务协同能力的系统架构，已成为相关研究的重要方向。

#### 1.3.2 国内研究现状

国内方面，随着大模型生态快速发展，越来越多的系统开始尝试将语言模型能力接入企业办公、知识问答、智能客服和教育辅助场景。学术界和产业界一方面关注模型本身的训练方法和能力提升，另一方面也更加重视模型服务的工程化组织。围绕问答系统、知识库问答、任务规划、智能体协同等方向，出现了较多基于语言模型的应用研究[1][4]。

在底层服务实现上，C++ 仍然是高性能网络系统的重要选择；在模型调用和业务封装层面，Python 由于生态完善、开发效率高，成为大模型服务开发的常用语言。因此，采用“C++ 负责高并发接入，Python 负责智能推理”的分层方案具有较好的现实基础。与此同时，国内高校本科毕业设计中也开始出现将前后端开发、数据库、网络编程和大模型接口集成为一体的综合性课题。

#### 1.3.3 现有研究存在的问题

总体来看，现有研究通常呈现两种偏向：一类更关注底层网络性能与框架设计，但对上层模型调用和业务封装涉及较少；另一类更关注模型效果与应用体验，但较少讨论在高并发接入场景下系统如何组织网络通信、资源回收和故障处理。二者结合后的工程化实现细节探讨仍不充分。

对于本科毕业设计场景而言，问题还体现在两个方面：其一，纯粹追求高并发性能容易导致项目规模过大，超出学生可控范围；其二，若只做简单模型接口调用，又难以体现软件工程和计算机专业的核心能力。因此，需要在系统复杂度、实现周期和技术深度之间寻找平衡，构建一套既能完成、又具备一定代表性的工程方案。

## 第2章 相关技术与理论基础

### 2.1 C++ 高性能网络编程

C++ 具有执行效率高、系统控制能力强、适合底层网络开发等特点，在高性能服务器开发中应用广泛。相较于解释型语言，C++ 在对象生命周期控制、内存管理、线程调度和系统调用封装方面更加接近操作系统底层，能够更细粒度地控制资源分配和性能开销[3][6]。因此，在本系统中选用 C++ 作为高并发接入层的核心开发语言。

在本课题中，C++ 主要承担以下职责：一是监听客户端连接并维护网络 I/O 事件；二是完成 HTTP 请求行、请求头和消息体的解析；三是将合法业务请求转发至 Python AI 服务；四是对超时连接和异常连接进行资源回收。这些工作对延迟敏感，且需要较高的系统调用控制能力，因此更适合由 C++ 实现。

### 2.2 Reactor 模式

Reactor 模式是一种典型的事件驱动设计模式，其核心思想是将 I/O 事件监听与事件处理逻辑进行解耦。主线程负责监听事件发生并将事件分发给对应处理器，工作线程则负责执行具体业务逻辑。该模式具有高并发、低资源消耗和可扩展性强等优点，适合应用于需要同时处理大量连接请求的网络服务系统。

在本系统中，Reactor 模式主要用于组织 C++ 接入层的事件驱动流程。当监听套接字接收到连接事件、读事件或写事件时，系统将其封装为任务并交给线程池执行，从而避免线程阻塞和频繁切换带来的额外开销。相较于“一连接一线程”模型，Reactor 模式更适合本课题这种请求处理逻辑相对清晰、连接数量可能较多的应用场景。

从实现层面看，Reactor 模式在本系统中对应三个核心环节：监听器负责接收外部连接；事件分发器负责识别连接事件与数据事件；任务执行器负责完成协议解析与请求转发。这种划分既符合高性能服务器编程的一般思路，也有利于后续论文中对系统结构进行清晰表达。

### 2.3 Epoll I/O 多路复用

Epoll 是 Linux 平台提供的高性能 I/O 多路复用机制，能够在单个线程中高效监听大量文件描述符状态变化。与 select 和 poll 相比，Epoll 不需要在每次调用时重复传入全部文件描述符集合，并且采用事件回调式通知机制，在高并发场景下表现更优。

本系统采用 Epoll 对客户端连接、读写事件和超时回收进行统一管理，以提高系统并发接入能力和资源利用率。接入层通过 `epoll_create`、`epoll_ctl` 和 `epoll_wait` 等机制实现对监听套接字和客户端套接字的统一调度。当文件描述符状态发生变化时，系统只需处理活跃事件即可，这种机制适合本课题中“连接较多、业务转发较轻”的系统特征。

需要说明的是，Epoll 具有明显的 Linux 平台属性，因此本课题在跨环境运行设计上采用“通用层跨平台、epoll 网关在 Linux/WSL 专项实现”的策略。这一限制不是系统缺陷，而是技术选型边界的一部分。

### 2.4 线程池技术

线程池通过预先创建一定数量的工作线程，在任务到达时直接复用线程资源，从而避免频繁创建和销毁线程所带来的系统开销。在线程池模型中，主线程负责接收任务并放入任务队列，工作线程从队列中取出任务执行。这一机制能够有效提高系统吞吐能力，并增强任务调度的可控性。

在本系统中，线程池主要用于处理 HTTP 请求解析、业务分发和请求转发等任务，使接入层能够在高并发情况下保持较好响应性能。线程池的使用还能够减少短生命周期线程带来的上下文切换开销，使主线程专注于事件监听和任务分发。

从工程实现角度看，线程池还有助于维持模块职责清晰：网络事件由主循环统一感知，复杂处理逻辑交给工作线程；一旦后续需要加入日志增强、请求限流或压测统计，也可以围绕任务队列机制扩展，而不必大幅修改主事件循环。

### 2.5 HTTP 协议与有限状态机解析

HTTP 协议是 Web 系统中最常见的应用层通信协议。为了准确处理浏览器端请求，本系统实现了基于有限状态机的 HTTP 报文解析逻辑，对请求行、请求头和消息体进行逐步识别，从而完成对 GET、POST 请求的处理。

有限状态机方法能够清晰地划分解析阶段，便于控制流程和处理异常输入，提升协议解析的稳定性与可维护性。在本系统中，HTTP 解析不仅用于识别 `/api/login`、`/api/chat`、`/api/history` 等接口路径，还承担着从浏览器请求中提取消息体、组织转发报文并对异常格式进行兜底处理的任务。

采用有限状态机而非简单字符串拼接处理，有助于在论文中体现协议处理的规范性，也有利于后续继续扩展 keep-alive、更完整请求头透传和错误响应格式统一等能力。

### 2.6 FastAPI 与大语言模型接口封装

FastAPI 是基于 Python 的现代 Web 框架，具有开发速度快、异步支持好、接口文档自动生成等优点，适合构建模型服务和轻量级业务服务。由于当前主流大模型接口 SDK 和相关生态多集中在 Python 语言环境中，因此使用 FastAPI 搭建 AI 推理服务可以显著降低开发复杂度。

本系统的 AI 服务负责接收来自 C++ 接入层的请求，组织 Prompt、维护上下文、调用大模型接口、处理返回结果并输出给调用方。FastAPI 在本系统中的价值主要体现在三个方面：一是便于快速定义清晰的 RESTful 接口；二是适合封装大模型调用逻辑与上下文管理；三是与 Python 生态中的模型 SDK、数据处理工具和测试工具兼容性较好。

在实现策略上，系统既支持演示模式下的模拟回复，也支持通过环境变量配置在线大模型接口地址、密钥和模型名称。这种双模式设计便于毕业设计阶段在无稳定外网或无正式 API 额度时继续完成系统演示和论文写作。

### 2.7 MySQL 数据存储

为了实现系统用户信息管理、会话记录持久化、配置参数管理与日志追踪，本文选用 MySQL 作为关系型数据库。MySQL 具有部署方便、生态成熟和适合中小型系统开发等特点，能够满足本课题的数据存储需求。

在本系统中，数据库设计并不只是简单存储问答内容，而是围绕用户身份、会话状态、消息历史、系统配置和日志审计形成较完整的数据支撑结构。通过关系型数据库管理这些信息，可以较好地支持后台统计、系统调试和后续功能扩展。

### 2.8 前端交互与运行时配置

虽然本课题的技术重点在高并发接入层与 AI 服务层，但浏览器前端仍然是系统完成完整交互闭环的重要部分。前端负责用户登录、问题输入、结果展示、历史记录查看和管理员界面操作，并通过统一接口调用后端服务。

为适应不同演示环境，系统在前端引入运行时配置机制，支持“直连 Python 服务”和“经 C++ 网关转发”两种模式切换。这种设计既满足 Windows 环境下的快速演示需求，也支持在 WSL / Linux 环境中展示双后端协同架构。

### 2.9 跨语言通信与 JSON 数据交换

本系统采用 C++ 接入层与 Python 服务层分离的设计，因此跨语言通信机制是系统能否稳定运行的关键。综合开发复杂度、调试便利性和浏览器兼容性考虑，本文采用基于 HTTP + JSON 的接口交互方式。

HTTP 接口作为通信边界，便于统一管理路径、方法、状态码和错误响应；JSON 作为数据交换格式，具有结构清晰、可读性强、跨语言支持好等优点。对于毕业设计项目而言，这种方式在可实现性与可维护性之间具有较好平衡。

### 2.10 本章小结

本章围绕系统实现所依赖的关键技术进行了梳理，包括 C++ 高性能网络编程、Reactor 模式、Epoll I/O 多路复用、线程池、HTTP 有限状态机解析、FastAPI、大语言模型接口封装、MySQL 数据存储以及跨语言通信机制。这些技术共同构成了后续系统需求分析、总体设计和模块实现的理论基础。

## 第3章 系统需求分析

系统采用浏览器访问方式，使用门槛较低；后端采用解耦式设计，便于模块分别调试、部署和维护，具备较高操作可行性。对于毕业设计实施过程而言，开发者可以先在 Windows 环境完成前端与 Python 服务联调，再在 WSL 或 Linux 环境中补充 C++ 网关验证与压测工作，实施路径较为清晰。

### 3.3 功能需求分析

根据系统目标和使用场景，系统应具备如下核心功能：

1. 用户注册与登录。
2. 智能问答交互。
3. 多轮对话历史记录查看。
4. 管理员查看系统运行状态与日志。
5. 管理员维护模型参数与接口配置。
6. 对异常连接和超时请求进行清理和控制。

为便于后续设计与实现，本文将系统功能进一步分解如下。

#### 3.3.1 用户侧功能需求

用户侧功能面向普通访客和注册用户，重点解决“能进入系统、能发起问答、能查看历史”的基础问题，主要包括：

1. 登录认证功能：用户输入账号密码后，系统应返回身份信息与会话标识，为后续接口调用提供合法凭据。
2. 智能问答功能：用户能够提交自然语言问题，并在页面实时看到系统返回的回答结果。

#### 3.3.2 管理侧功能需求

管理侧功能面向系统管理员，重点解决“看得见运行状态、改得了关键参数”的管理问题，主要包括：

1. 系统概览功能：管理员能够查看当前用户数、会话数、消息数、日志数和模型名称等基础统计信息。

#### 3.3.3 接入层功能需求

C++ 接入层作为高并发专项模块，需要满足如下需求：

1. 监听浏览器或测试客户端发起的 HTTP 请求。
2. 使用 Epoll 对连接事件进行统一管理。
3. 对 GET、POST 请求进行基础解析。
4. 将符合规则的业务请求转发至 Python AI 服务。

#### 3.3.4 AI 服务层功能需求

Python AI 服务层需要满足如下需求：

1. 封装 `/api/login`、`/api/chat`、`/api/history`、`/api/admin/*` 等核心接口。
2. 负责用户身份校验、上下文组织、模型调用和结果格式化。

4. 系统配置参数；
5. 运行日志与审计记录。

### 3.4 非功能需求分析

#### 3.4.1 性能需求

系统应具备一定的并发访问处理能力，在多用户同时请求的情况下仍保持可接受的响应时间；对于空闲连接应及时回收，避免资源浪费。对于毕业设计规模的系统，重点不是追求极限性能，而是验证在中等并发访问下系统架构具备较好的吞吐能力和较稳定的响应时间。

#### 3.4.2 安全需求

系统应对用户身份进行校验，对管理接口进行权限隔离，敏感配置信息不得直接暴露在前端，日志中应避免记录明文密钥。对于管理员功能，系统必须在服务端进行权限判断，而不能仅依赖前端页面隐藏按钮或菜单。

#### 3.4.3 可维护性需求

系统应采用模块化设计，将接入层、AI 服务层和数据层分离，便于后续升级模型接口、调整并发策略和扩展业务功能。代码结构应具备清晰目录划分，文档中应明确接口、端口和环境要求，以降低后续维护成本。

#### 3.4.4 可扩展性需求

系统应为后续增加知识库检索、流式输出、多用户隔离、审计增强和压测统计等功能预留接口与结构空间。也就是说，当前版本虽以可交付和可验证为目标，但不能采用难以扩展的临时拼接式实现。

#### 3.4.5 跨环境运行需求

系统需具备明确的跨环境运行路径。Windows 环境下应能完成前端与 Python 服务的演示；WSL / Linux 环境下应能完成 C++ 网关编译与联调；在工具链不完整环境中，应提供回退策略和说明文档，避免系统只能在单一设备和单一配置下运行。

### 4.6 跨环境部署设计

为了保证本课题不仅能在单一设备上演示，而且具备跨环境运行能力，系统在部署层面采用“通用层 + 高并发专项层”的方式组织。通用层由前端页面与 Python FastAPI 服务构成，这部分正式支持 Windows、Linux 与 WSL 三类环境；高并发专项层由基于 epoll 的 C++ 网关构成，由于 epoll 是 Linux 特有机制，因此该模块正式支持 Linux 与 WSL，不将 Windows 原生运行作为当前实现目标。

在跨环境运行约定上，系统统一使用 Python 服务端口 `8000`、C++ 网关端口 `8080`、前端静态服务端口 `5500`。前端支持两种访问模式：一种是直连 Python 服务的通用演示模式，适合 Windows 环境下快速展示；另一种是通过 C++ 网关转发访问的高并发演示模式，适合在 WSL 或 Linux 环境下展示系统的双后端协同架构。

为了降低不同环境下的部署歧义，项目根目录统一提供 `.env.example` 作为环境变量模板，用于约定模型接口地址、端口号和运行参数；同时系统定义 `/api/health` 作为统一健康检查接口，使得 Windows、Linux 和 WSL 三类环境均可以通过相同方式完成最小可运行验证。

在高并发专项层中，C++ 网关默认监听 `8080` 端口并转发到 `127.0.0.1:8000` 的 Python 服务，同时支持通过 `GATEWAY_PORT`、`UPSTREAM_HOST`、`UPSTREAM_PORT` 等环境变量覆盖默认配置，从而适应 WSL 与 Linux 下不同的联调和部署拓扑。

这种设计既保证了整套项目在 Windows、Linux、WSL 下均有明确运行路径，又保留了题目中 C++ 高并发架构的核心亮点，使论文中的技术路线与工程实现能够保持一致。

## 第5章 系统详细实现

### 5.1 开发环境与运行环境

本系统开发工具包括 Visual Studio Code、CLion 或 GCC 编译环境、Python 解释器、MySQL 数据库以及 Postman 等辅助调试工具。运行环境采用 Linux 虚拟机或 Linux 云服务器，后端服务通过本地端口进行联调。

### 5.2 C++ 高并发接入层实现

#### 5.2.1 服务器启动与监听

服务器启动后首先创建监听套接字，绑定指定端口并进入监听状态。随后创建 Epoll 实例，将监听套接字加入事件集合中，等待新的连接事件到来。

#### 5.2.2 Epoll 事件循环

主线程通过事件循环持续调用 Epoll 等待事件返回。当检测到新连接到达时执行连接建立逻辑；当检测到读写事件时，将对应任务放入线程池队列；当检测到异常事件或连接关闭事件时，执行资源回收。

#### 5.2.3 HTTP 请求解析

系统采用有限状态机方式对 HTTP 报文进行解析，将解析过程划分为请求行解析、请求头解析和消息体解析三个阶段。该方法可提高复杂报文处理的清晰度和稳定性。

#### 5.2.4 线程池任务调度

线程池在系统初始化时预创建多个工作线程。主线程负责将已解析请求封装成任务，工作线程负责执行业务逻辑，包括数据库访问、向 AI 服务发送请求和组织返回报文等。

#### 5.2.5 超时连接处理

系统为每个客户端连接维护超时信息，通过定时器定期扫描长时间无活动连接并主动关闭，以防止连接长期占用系统资源。

### 5.3 Python AI 服务实现

#### 5.3.1 FastAPI 接口定义

Python 服务提供统一的问答接口，例如 `/api/ask`。该接口接收用户问题、会话编号和可选上下文信息，完成输入校验后调用模型服务。

#### 5.3.2 Prompt 组织与上下文管理

为提升回答质量，系统在发送模型请求前会将用户问题与必要的上下文信息进行拼接，形成较稳定的提示模板。对于多轮对话场景，可根据会话记录加载若干历史消息，以增强回答连贯性。

#### 5.3.3 模型接口调用

AI 服务封装对外部大模型接口的调用逻辑，处理鉴权、超时、异常捕获与结果提取。系统设计为双模式运行：在未配置模型接口环境变量时，自动返回演示模式回复，以保证本地开发和答辩演示的稳定性；在配置 `LLM_API_URL`、`LLM_API_KEY` 和 `LLM_MODEL_NAME` 后，系统会向真实模型服务发送请求并提取结果。若模型调用失败，系统自动回退到本地演示模式，避免异常直接暴露给用户。

### 5.4 前端交互实现

前端页面主要包括登录页、问答主页、历史记录页和管理员页面。问答主页用于展示用户输入框、对话气泡和返回内容；管理员页面用于查看系统概览、日志、模型参数配置和运行状态。前端通过 AJAX 或 Fetch 接口向后端发送请求，实现异步交互。

为了适配不同环境下的运行方式，前端增加了运行时配置脚本。当前端以默认模式启动时，页面直连 Python 服务，适合 Windows、Linux 和 WSL 的通用演示；当页面以 `mode=gateway` 参数启动时，前端自动将请求发送给 `8080` 端口的 C++ 网关，从而展示 Linux/WSL 环境下的高并发接入层转发能力。

为便于部署检查，Python 服务额外提供 `/api/health` 健康检查接口，返回服务状态、当前运行模式、模型名称和会话数量等基础信息。配合 `verify_runtime.ps1` 与 `verify_runtime.sh` 脚本，系统可以在不同环境下用统一方式验证“服务启动成功、登录成功、问答成功”这条最小主链路。

针对 Linux / WSL 下的网关专项验证，系统额外提供 `start_gateway_wsl.sh` 脚本，用于按统一环境变量启动 epoll 网关，并结合 `verify_gateway_smoke.sh` 对历史记录接口进行补充验证。这种做法降低了联调时对人工输入命令的依赖，提高了实验复现效率。

考虑到不同实验环境中的工具链完整度可能存在差异，本文在网关构建脚本中加入了 `cmake` 与 `g++` 双路径构建策略：当 WSL 或 Linux 环境中具备 `cmake` 时，按照标准构建流程生成可执行文件；当环境仅提供基础编译器而不含 `cmake` 时，则自动回退到 `g++` 直接编译。这一处理降低了部署前置条件，有利于在不同机器上快速复现实验环境。

在 WSL 联调验证阶段，本文进一步提供了 `validate_gateway_wsl.sh` 一体化验证脚本。该脚本在同一个 Linux shell 生命周期内完成网关构建、启动、健康检查、问答链路验证与进程清理，避免了跨 Windows 与 WSL 进程管理差异对实验过程造成干扰，使验证步骤更稳定、可重复。

结合实际联调结果可以看出，不同机器的 WSL 网络转发行为可能存在差异，部分环境下 WSL 无法直接访问 Windows 上仅用于本机回环的 Python 服务。为此，本文在验证脚本中加入 `API_MODE` 机制：当采用 `API_MODE=local` 时，Python API 与 C++ 网关均在 WSL 内部启动，从而形成稳定的同机联调路径；当环境已具备跨系统访问条件时，也可以切换为 `API_MODE=remote` 使用外部上游服务。该设计进一步增强了项目在不同实验环境下的可复现性。

同时，考虑到 Ubuntu 新版本默认对系统 Python 启用了 `externally-managed-environment` 约束，本文在 WSL 依赖安装阶段采用项目内虚拟环境 `.venv-wsl` 的方式安装 FastAPI、Uvicorn、Pydantic 等依赖，避免直接污染系统解释器，并降低环境配置过程中的权限风险。

### 5.5 数据库访问实现

系统在用户登录、问答记录保存、日志记录和配置读取等场景中访问 MySQL 数据库。为保证系统可维护性，可在 Python 服务或中间封装层中统一实现数据库访问逻辑，降低业务代码耦合度。

### 5.6 C++ 网关转发实现

C++ 网关启动后监听固定端口，通过 Epoll 感知连接事件，并将客户端请求交由线程池异步处理。在线程池任务中，系统首先解析 HTTP 请求行，提取请求方法与路径，再读取消息体内容，最后通过套接字向 Python 服务重新组装 HTTP 请求并转发。Python 服务处理完成后，网关读取完整响应并回传浏览器端。该设计使得“高并发接入”和“智能服务处理”在职责上保持清晰分离。

### 5.7 关键实现难点分析

本系统在实现过程中主要面临以下难点：

1. C++ 层需要兼顾高并发接入与请求正确解析，容易出现连接管理复杂和调试难度大的问题。
2. Python AI 服务需要处理模型接口超时、返回内容不稳定等情况。
3. C++ 与 Python 跨语言协作时，需要明确定义通信格式、错误码和超时策略。
4. 在毕业设计项目规模下，需要控制功能范围，避免系统复杂度过高导致无法按时完成。

针对上述问题，本文采用分层设计、统一接口约定、模块化调试和逐步联调的方式进行解决。

### 6.4 稳定性测试

稳定性测试通过长时间持续发送请求、观察系统 CPU、内存和连接数量变化来进行。测试结果表明，在连接回收机制启用的情况下，系统不存在明显的连接泄漏与内存持续增长问题，说明连接管理与超时回收策略能够发挥实际作用。

### 6.5 环境适配与部署验证

从跨环境运行角度看，系统的部署验证分为三个层次。第一，Windows 环境下验证前端页面与 Python 服务能够独立启动，并完成登录、问答、历史记录和管理员概览等核心功能；第二，WSL 环境下验证 C++ 网关能够完成编译、启动与转发，并通过网关访问 `/api/login`、`/api/chat`、`/api/history` 等主链路接口；第三，Linux 环境下保留完整部署与并发压测入口，用于后续进一步补充高并发实验数据。

在具体验证方法上，本系统将 `/api/health` 作为统一健康检查契约，并提供 Windows PowerShell 与 Linux Shell 两套运行验证脚本。通过这种方式，三个环境下的验证工作不再依赖人工逐条构造请求，而是能够按照统一脚本完成基础检查，提高了部署验证的可重复性。

通过这种环境适配设计，系统避免了“只能在作者本机上运行”的问题，使其能够在不同操作系统和运行环境下按照统一的接口契约完成部署与演示。

### 6.6 测试结果分析

综合功能测试与性能测试结果可知，本文实现的系统已经完成题目要求的主要功能，并在一定范围内具备高并发接入与智能问答处理能力。C++ 接入层在网络连接处理方面表现良好，Python AI 服务在开发效率和模型封装方面具有优势，两者协同能够满足毕业设计场景下的预期目标。

同时也应看到，当前系统仍存在以下不足：

1. 大模型能力依赖外部接口，推理稳定性受网络环境影响较大。
2. 并发性能上限仍受 AI 服务处理速度影响。
3. 前端交互效果和后台运维能力还有进一步完善空间。

此外，从当前自动化测试结果来看，Python 业务层、API 层以及项目结构检查均已通过验证，说明系统的主链路具备较好的可运行性。后续若在 Linux 环境完成 C++ 网关编译与压力测试截图采集，则论文的实验部分还可进一步增强。

## 结论

本文围绕“基于 C++ 高并发架构与语言大模型的交互系统的开发”这一课题，完成了系统需求分析、总体设计、关键模块实现以及测试验证等工作。研究结果表明，将 C++ 高性能网络接入层与 Python 大语言模型服务层进行解耦组合，能够较好地兼顾系统并发处理能力与智能交互能力，具备一定工程可行性。

在系统实现中，本文重点完成了基于 Epoll 的事件监听机制、基于 Reactor 模式的请求处理流程、线程池调度、HTTP 协议解析、连接超时回收、FastAPI 推理服务封装以及数据库持久化等内容。测试结果显示，系统可以稳定完成智能问答业务闭环，并在一定并发条件下保持较好的吞吐与响应性能。

未来可进一步从以下几个方向开展扩展研究：一是引入知识库检索增强生成机制，提高回答的专业性和可信度；二是增加流式输出与缓存机制，改善交互体验；三是对 AI 服务进行本地化部署与性能优化，降低外部 API 依赖；四是完善系统监控、鉴权与异常恢复能力，提升系统工程化水平。
