# 基于C++高并发架构与语言大模型的交互系统的开发


## 摘要

随着大语言模型在智能问答、企业知识服务、在线咨询和辅助决策等场景中的广泛应用，面向真实业务环境的智能交互系统不仅需要具备较强的语义理解与文本生成能力，还需要具备较高的并发处理能力、稳定性和可扩展性。传统以脚本语言为主的智能问答服务在快速开发方面具有明显优势，但在高并发网络连接管理、请求调度和资源控制方面往往存在性能瓶颈。针对上述问题，本文设计并实现了一种基于 C++ 高并发服务架构与语言大模型推理服务相结合的智能交互系统。

系统整体采用前后端分离与双后端微服务协同的设计思路。前端通过 Web 页面完成用户交互；接入层采用基于 Linux 平台的 C++ 高性能服务器，利用 Reactor 模式、Epoll I/O 多路复用、线程池、HTTP 有限状态机解析以及连接超时管理等机制，实现高并发连接接入与请求调度；业务智能层采用 Python FastAPI 框架封装大语言模型接口，负责用户语义请求处理、上下文拼接、结果生成与数据返回；数据层使用 MySQL 完成用户信息、会话记录、系统配置与日志等内容的持久化存储。系统通过 HTTP 接口完成 C++ 接入层与 Python AI 服务之间的跨语言通信，在保持底层高性能的同时兼顾上层智能服务开发效率。

在系统实现基础上，本文从功能完整性、并发性能、稳定性和可维护性等方面展开测试。测试结果表明，该系统能够完成用户登录、发起智能问答、记录会话、后台配置管理等核心业务流程，在一定并发压力下能够保持较好的吞吐能力和响应时间，证明了“底层高性能网络架构 + 上层语言大模型能力”相结合的方案具有较强的工程可行性和应用价值。

本文的研究工作对于高校软件工程类毕业设计具有一定参考意义，也为后续进一步引入知识库检索、流式输出、权限隔离和模型服务优化等功能提供了可扩展基础。

**关键词：** C++；高并发；Reactor；Epoll；FastAPI；大语言模型；智能交互系统

## Abstract

With the rapid development of large language models, intelligent interaction systems are being widely used in enterprise knowledge services, online consultation, educational assistance, and data analysis scenarios. In practical applications, such systems are required not only to provide accurate natural language understanding and response generation, but also to support high concurrency, stable operation, and scalable deployment. Script-based AI services are efficient for rapid development, yet they often encounter bottlenecks in network connection management, request scheduling, and system resource control under high-load conditions. To address these issues, this thesis designs and implements an interactive system based on a high-concurrency C++ architecture combined with a large language model service.

The system adopts a front-end and back-end separated architecture with two cooperative backend services. The front-end provides browser-based interaction. The access layer is implemented with a Linux-based C++ high-performance server, which uses the Reactor pattern, Epoll I/O multiplexing, thread pools, HTTP finite-state-machine parsing, and connection timeout management to support high-concurrency request processing. The intelligent business layer is implemented with Python FastAPI, which encapsulates large language model APIs for semantic request processing, context organization, and answer generation. MySQL is used for persistent storage of user information, dialogue records, system parameters, and logs. Cross-language communication between the C++ service and the Python AI service is achieved through HTTP interfaces, balancing low-level performance with upper-layer development flexibility.

Based on the implemented system, this thesis performs functional testing, concurrency testing, stability testing, and maintainability analysis. The results show that the system can complete core processes such as user login, intelligent question answering, dialogue recording, and background configuration management. Under a certain concurrent load, the system maintains acceptable throughput and response latency, which verifies the engineering feasibility of combining a high-performance network framework with large language model capabilities.

This work provides a practical reference for undergraduate software engineering projects and establishes a foundation for future extensions such as knowledge retrieval enhancement, streaming output, access control, and model service optimization.

**Key words:** C++; high concurrency; Reactor; Epoll; FastAPI; large language model; interactive system

## 第1章 绪论

### 1.1 研究背景与意义

近年来，以 ChatGPT、DeepSeek、通义千问、智谱 GLM 等为代表的大语言模型快速发展，推动自然语言处理技术由传统规则驱动和浅层机器学习阶段迈向以预训练模型为核心的智能化阶段。与传统问答系统相比，大语言模型在上下文理解、复杂语义表达、开放域知识组织以及多轮对话生成方面表现更加突出，已逐渐应用于智能客服、办公助手、知识库问答、教育辅导和行业咨询等场景。

但是，从软件系统工程角度看，模型能力的提升并不等同于应用系统成熟度的提升。一个面向真实使用场景的智能交互平台不仅要能够生成自然语言回答，还需要具备稳定的网络接入能力、清晰的请求处理链路、可维护的数据记录机制以及必要的后台管理能力。若直接采用单体式脚本服务承载浏览器接入、请求解析、模型调用、历史记录和管理功能，在多用户同时访问时容易暴露连接管理粗糙、资源回收不及时、请求链路阻塞和日志追踪困难等问题。

高并发接入问题在 Web 应用、即时通信、在线考试和智能服务平台中具有共性。C++ 在底层网络开发、系统调用控制和运行效率方面具有优势，适合承担连接监听、I/O 多路复用和请求转发等基础工作；Python 在大模型接口封装、业务逻辑组织和快速开发方面生态成熟，适合承担智能服务层任务。因此，将 C++ 高并发接入层与 Python 大模型服务层进行分层协同，是兼顾性能、开发效率和可维护性的一种可行方案。

本课题以“基于 C++ 高并发架构与语言大模型的交互系统的开发”为研究对象，设计并实现一套面向浏览器问答场景的智能交互系统。系统通过前端页面完成用户交互，通过 C++ 网关完成高并发连接接入与请求转发，通过 Python FastAPI 服务完成身份校验、上下文组织、模型调用和结果返回，并通过数据库保存用户、会话、消息、配置和日志等数据。该方案既体现高性能网络编程能力，又能结合大语言模型应用开发，具有较强的工程实践意义。

本研究的意义主要体现在以下三个方面。第一，在理论层面，探索高性能网络架构与大语言模型服务化部署的结合方式，为“底层高性能接入 + 上层智能能力封装”的系统设计提供分析样本。第二，在工程层面，验证 C++ 接入层与 Python AI 服务协同设计的可行性，为类似智能交互系统的模块划分、接口组织和运行验证提供参考。第三，在实践层面，形成一套适合本科毕业设计实施的完整方案，包括需求分析、概要设计、系统实现、测试验证和论文撰写支撑。

### 1.2 国内外研究现状

在国外研究与工程实践中，高性能服务器领域较早开展了事件驱动网络模型、I/O 多路复用机制和高并发服务端编程研究。Epoll、Kqueue、Reactor、Proactor 等模型在互联网服务端开发中已有广泛应用，相关实践表明，事件驱动模型在高连接数场景下通常比简单的一连接一线程模型具有更好的资源利用率和响应能力。与此同时，FastAPI、gRPC、异步消息队列、容器化部署和模型网关等技术也被广泛用于智能服务系统建设，使大模型能力能够以接口形式接入实际业务系统。

在人工智能服务部署方面，国外研究较早关注模型服务化、推理加速和 AI 基础设施优化。随着大模型应用从研究验证走向业务落地，系统架构不再只关注模型效果本身，还需要考虑请求调度、接口稳定性、上下游服务隔离、异常降级和部署复现等工程问题。对于智能问答类应用而言，如何将模型调用能力放入一个稳定、可扩展、可监控的服务体系中，已经成为大模型应用工程化的重要方向。

国内方面，随着大模型生态快速发展，越来越多的系统开始尝试将语言模型能力接入企业办公、知识问答、智能客服和教育辅助场景。学术界和产业界一方面关注模型本身的训练方法和能力提升，另一方面也更加重视模型服务的工程化组织。围绕问答系统、知识库问答、智能体协同和业务辅助决策等方向，已经出现较多基于语言模型的应用研究与项目实践。

在底层服务实现上，C++ 仍然是高性能网络系统的重要选择；在模型调用和业务封装层面，Python 由于生态完善、开发效率高，成为大模型服务开发的常用语言。因此，采用“C++ 负责高并发接入，Python 负责智能推理”的分层方案具有较好的现实基础。对于本科毕业设计而言，该方案能够同时覆盖前后端协同、数据库设计、网络编程、接口封装和系统测试等多个核心能力点，避免课题内容过于单一。

现有研究和实践仍存在一定不足：一类工作更关注底层网络性能与框架设计，但对大模型调用和业务封装涉及较少；另一类工作更关注模型效果和应用体验，但较少讨论高并发接入场景下的连接管理、资源回收和故障处理。二者结合后的工程化实现细节仍有进一步梳理空间。本课题正是在这一背景下，尝试构建一套规模可控、功能闭环明确、技术路线清晰的智能交互系统。

### 1.3 主要研究内容与论文组织结构

围绕课题目标，本文主要完成以下研究内容：第一，分析高并发智能交互系统的业务特点、用户角色、功能需求和非功能需求；第二，设计前端页面、C++ 高并发接入层、Python FastAPI 智能服务层和数据库层构成的系统总体架构；第三，设计用户登录、智能问答、历史记录、后台概览、日志查看和配置维护等核心功能模块；第四，设计用户、会话、消息、日志和配置等数据库表结构；第五，设计系统接口、业务流程和跨环境运行方案，为后续实现与测试提供依据。

在技术路线上，本文以浏览器端问答场景为牵引，从需求分析出发，逐步形成“Web 前端 - C++ 接入层 - Python 智能服务层 - MySQL 数据层”的四层结构。浏览器通过 HTTP 发起登录、问答、历史记录和后台管理请求；C++ 网关负责连接管理、协议解析和请求转发；Python 服务负责用户校验、上下文组织、模型调用、日志记录和数据返回；数据库负责对关键业务数据进行持久化存储。系统同时支持 Windows 下的前端与 Python 服务直连演示，以及 Linux / WSL 下经 C++ 网关转发的双后端协同演示。

本课题拟重点解决以下问题：高并发接入层与智能业务层的职责边界如何划分；多轮问答、历史记录、配置管理和日志追踪等功能如何形成统一业务闭环；C++ 网关与 Python 服务之间如何通过 HTTP + JSON 建立清晰接口契约；系统如何在 Windows、Linux、WSL 等不同环境下给出明确运行路径，降低只能在单一设备演示的风险。

全文组织结构如下。第1章为绪论，介绍研究背景与意义、国内外研究现状、主要研究内容和论文组织结构。第2章为需求分析，分析系统功能需求、非功能需求、可行性、关键业务流程和用例。第3章为概要设计，给出系统总体架构、功能模块、数据库结构、接口设计和界面设计。后续章节将在完整论文中继续展开系统详细实现、测试分析、结论、参考文献和附录等内容。


## 第2章 系统需求分析

### 2.1 系统目标

本系统面向智能问答与业务辅助交互场景，构建一个具备高并发接入能力与语言模型智能处理能力的交互系统。系统应在浏览器端为用户提供统一入口，在服务器端完成连接管理、请求解析、消息转发和智能回复生成，并能够支持管理员对系统运行状态和关键参数进行管理。

从功能目标上看，系统需要完成“登录 - 提问 - 返回答案 - 保存历史 - 管理配置”的完整业务闭环；从性能目标上看，系统需要具备一定的并发接入能力，能够在多请求场景下保持较为稳定的响应；从工程目标上看，系统应具有清晰的模块边界和跨环境运行路径，便于测试、演示和后续扩展。

### 2.2 可行性分析

#### 2.2.1 技术可行性

本系统所依赖的关键技术均较为成熟。Linux、C++ Socket 编程、Epoll、线程池、FastAPI、MySQL 等技术已有大量工程实践，大模型接口调用方式也相对清晰，因此系统具备较高技术可行性。

在底层网络实现上，Epoll 与 Reactor 模式适合支撑高并发接入实验；在业务层实现上，FastAPI 可快速搭建可测试的接口服务；在数据层实现上，MySQL 的表结构设计和访问方式相对成熟。当前项目已完成代码骨架与主要接口设计，也为技术可行性提供了初步验证基础。

#### 2.2.2 经济可行性

系统开发主要基于开源工具与通用开发环境完成，不需要昂贵商业软件授权。开发工具主要包括 VS Code、Visual Studio、PyCharm、GCC 或 WSL 环境、MySQL 和浏览器等。若采用在线大模型 API，可利用学生开发额度或低成本按量调用方式完成测试；在无外部接口条件下，也可以切换到演示模式完成功能联调和答辩展示，因此整体成本可控。

#### 2.2.3 操作可行性

系统采用浏览器访问方式，使用门槛较低；后端采用解耦式设计，便于模块分别调试、部署和维护，具备较高操作可行性。对于毕业设计实施过程而言，开发者可以先在 Windows 环境完成前端与 Python 服务联调，再在 WSL 或 Linux 环境中补充 C++ 网关验证与压测工作，实施路径较为清晰。

### 2.3 功能需求分析

根据系统目标和使用场景，系统应具备如下核心功能：

1. 用户注册与登录。
2. 智能问答交互。
3. 多轮对话历史记录查看。
4. 管理员查看系统运行状态与日志。
5. 管理员维护模型参数与接口配置。
6. 对异常连接和超时请求进行清理和控制。

为便于后续设计与实现，本文将系统功能进一步分解如下。

#### 2.3.1 用户侧功能需求

用户侧功能面向普通访客和注册用户，重点解决“能进入系统、能发起问答、能查看历史”的基础问题，主要包括：

1. 登录认证功能：用户输入账号密码后，系统应返回身份信息与会话标识，为后续接口调用提供合法凭据。
2. 智能问答功能：用户能够提交自然语言问题，并在页面实时看到系统返回的回答结果。
3. 历史记录查看功能：用户能够查看当前会话内已保存的多轮问答内容，便于回顾上下文。
4. 会话标题生成功能：系统能够依据首次提问内容自动生成简要会话标题，提升交互可读性。

#### 2.3.2 管理侧功能需求

管理侧功能面向系统管理员，重点解决“看得见运行状态、改得了关键参数”的管理问题，主要包括：

1. 系统概览功能：管理员能够查看当前用户数、会话数、消息数、日志数和模型名称等基础统计信息。
2. 日志查看功能：管理员能够查看登录、问答、配置修改等关键事件的日志记录。
3. 配置维护功能：管理员能够调整模型名称、网关超时等关键参数。
4. 状态校验功能：管理员能够通过健康检查接口快速确认后端服务是否正常运行。

#### 2.3.3 接入层功能需求

C++ 接入层作为高并发专项模块，需要满足如下需求：

1. 监听浏览器或测试客户端发起的 HTTP 请求。
2. 使用 Epoll 对连接事件进行统一管理。
3. 对 GET、POST 请求进行基础解析。
4. 将符合规则的业务请求转发至 Python AI 服务。
5. 对异常请求、不支持方法、上游不可达等情况返回明确错误信息。

#### 2.3.4 AI 服务层功能需求

Python AI 服务层需要满足如下需求：

1. 封装 `/api/login`、`/api/chat`、`/api/history`、`/api/admin/*` 等核心接口。
2. 负责用户身份校验、上下文组织、模型调用和结果格式化。
3. 支持演示模式与真实模型模式切换。
4. 提供 `/api/health` 健康检查接口，便于联调与部署验证。

#### 2.3.5 数据管理功能需求

数据层需要支持以下内容的存储与读取：

1. 用户基础信息；
2. 会话信息与会话标题；
3. 多轮问答消息；
4. 系统配置参数；
5. 运行日志与审计记录。

### 2.4 非功能需求分析

#### 2.4.1 性能需求

系统应具备一定的并发访问处理能力，在多用户同时请求的情况下仍保持可接受的响应时间；对于空闲连接应及时回收，避免资源浪费。对于毕业设计规模的系统，重点不是追求极限性能，而是验证在中等并发访问下系统架构具备较好的吞吐能力和较稳定的响应时间。

#### 2.4.2 安全需求

系统应对用户身份进行校验，对管理接口进行权限隔离，敏感配置信息不得直接暴露在前端，日志中应避免记录明文密钥。对于管理员功能，系统必须在服务端进行权限判断，而不能仅依赖前端页面隐藏按钮或菜单。

#### 2.4.3 可维护性需求

系统应采用模块化设计，将接入层、AI 服务层和数据层分离，便于后续升级模型接口、调整并发策略和扩展业务功能。代码结构应具备清晰目录划分，文档中应明确接口、端口和环境要求，以降低后续维护成本。

#### 2.4.4 可扩展性需求

系统应为后续增加知识库检索、流式输出、多用户隔离、审计增强和压测统计等功能预留接口与结构空间。也就是说，当前版本虽以可交付和可验证为目标，但不能采用难以扩展的临时拼接式实现。

#### 2.4.5 跨环境运行需求

系统需具备明确的跨环境运行路径。Windows 环境下应能完成前端与 Python 服务的演示；WSL / Linux 环境下应能完成 C++ 网关编译与联调；在工具链不完整环境中，应提供回退策略和说明文档，避免系统只能在单一设备和单一配置下运行。

## 第3章 系统概要设计

### 3.1 系统设计原则

为保证系统满足毕业设计实现与论文表达双重需求，在总体设计中遵循以下原则：

1. 分层解耦原则：接入层、AI 业务层、数据层彼此分离。

3. 智能服务层：Python FastAPI 服务，负责 Prompt 组织、大模型接口调用和结果处理。

### 3.7 界面设计

系统界面采用浏览器访问方式，主要包括普通用户端和管理员端两类页面。普通用户端由登录区域、问答区域和历史记录区域组成：登录区域用于输入账号密码并展示当前运行模式；问答区域用于输入问题、显示多轮对话内容和当前会话标题；历史记录区域用于查看当前用户已保存的问答记录。该设计能够满足中期阶段“登录 - 提问 - 返回答案 - 保存历史”的核心演示链路。

管理员端页面主要包含系统概览、日志查看和配置维护三部分。系统概览展示用户数、会话数、消息数、日志数和模型名称等指标；日志查看用于观察登录、问答和配置修改等事件；配置维护用于调整模型名称、超时参数等运行配置。管理员页面通过服务端权限校验控制访问，避免只依赖前端页面隐藏入口。

前端界面在运行模式上支持直连 Python 服务和经 C++ 网关转发两种方式。用户可通过 URL 参数或运行时配置切换访问路径，从而在 Windows 环境下快速演示通用功能，在 WSL / Linux 环境下展示 C++ 高并发网关参与请求转发的系统结构。

### 3.8 跨环境部署设计

为了保证本课题不仅能在单一设备上演示，而且具备跨环境运行能力，系统在部署层面采用“通用层 + 高并发专项层”的方式组织。通用层由前端页面与 Python FastAPI 服务构成，这部分正式支持 Windows、Linux 与 WSL 三类环境；高并发专项层由基于 epoll 的 C++ 网关构成，由于 epoll 是 Linux 特有机制，因此该模块正式支持 Linux 与 WSL，不将 Windows 原生运行作为当前实现目标。

在跨环境运行约定上，系统统一使用 Python 服务端口 `8000`、C++ 网关端口 `8080`、前端静态服务端口 `5500`。前端支持两种访问模式：一种是直连 Python 服务的通用演示模式，适合 Windows 环境下快速展示；另一种是通过 C++ 网关转发访问的高并发演示模式，适合在 WSL 或 Linux 环境下展示系统的双后端协同架构。

为了降低不同环境下的部署歧义，项目根目录统一提供 `.env.example` 作为环境变量模板，用于约定模型接口地址、端口号和运行参数；同时系统定义 `/api/health` 作为统一健康检查接口，使得 Windows、Linux 和 WSL 三类环境均可以通过相同方式完成最小可运行验证。

在高并发专项层中，C++ 网关默认监听 `8080` 端口并转发到 `127.0.0.1:8000` 的 Python 服务，同时支持通过 `GATEWAY_PORT`、`UPSTREAM_HOST`、`UPSTREAM_PORT` 等环境变量覆盖默认配置，从而适应 WSL 与 Linux 下不同的联调和部署拓扑。

这种设计既保证了整套项目在 Windows、Linux、WSL 下均有明确运行路径，又保留了题目中 C++ 高并发架构的核心亮点，使论文中的技术路线与工程实现能够保持一致。
