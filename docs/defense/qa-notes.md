# 毕业设计答辩 Q&A 备忘

## 1. 为什么使用 C++ 网关

C++ 更适合实现底层网络接入、连接管理和高并发 I/O。本文使用 C++ 实现 epoll 网关，是为了体现高并发网络编程能力，同时把连接接入、HTTP 请求读取、上游转发和异常响应这些偏底层的工作从 Python 服务中拆出来。

答辩表达要点：

1. C++ 负责高并发接入，不负责复杂业务。
2. Python 负责模型调用和业务接口，避免用 C++ 重写所有上层逻辑。
3. 双后端拆分能体现工程分层和技术取舍。

## 2. 为什么 Python 服务适合承载模型调用

Python 生态中 FastAPI、HTTP 客户端、数据处理和模型 SDK 支持成熟，开发效率高，适合封装大语言模型接口、组织 Prompt、管理会话和返回 JSON 数据。模型服务通常受网络和推理耗时影响较大，因此将其放在 Python 层不会削弱 C++ 网关在连接接入层的价值。

## 3. 为什么 epoll 不承诺 Windows 原生运行

epoll 是 Linux 内核提供的 I/O 多路复用机制，Windows 原生环境没有 epoll。本文采用现实工程策略：整套项目支持 Windows、Linux、WSL；其中前端和 Python 服务属于跨平台通用层，C++ epoll 网关属于 Linux/WSL 高并发专项层。这不是缺陷，而是对底层机制和操作系统边界的清晰定义。

## 4. SQLite 与 MySQL 的关系

SQLite 是本地可运行的持久化验证实现，主要用于演示、测试和跨环境快速启动。MySQL 表结构体现在 `db/schema.sql` 和论文数据库设计中，作为正式数据库设计依据。两者的关系是：SQLite 保证项目可直接运行，MySQL 体现毕业设计中的正式数据库建模能力。

## 5. 多模型调用不等于多 Agent

多模型调用只是同时或轮流请求多个 LLM 接口，例如 DeepSeek、ChatGPT、Gemini 等。多 Agent 协作还需要明确的角色分工、任务拆解、消息传递、共享状态、结果评审和协调策略。因此，接入多个模型只能说具备多模型能力基础，不能直接等同于多 Agent 系统。

如果后续要扩展为多 Agent，可以增加以下模块：

1. 任务规划 Agent：负责把用户目标拆成步骤。
2. 执行 Agent：负责调用工具或模型完成子任务。
3. 评审 Agent：负责检查结果质量和一致性。
4. 协调器：负责维护状态、调度 Agent、合并输出。

## 6. 当前系统如何证明可运行

可运行性由三类证据支撑：

1. 接口证据：`/api/health`、`/api/login`、`/api/chat`、`/api/history` 等接口可访问。
2. 自动化测试证据：`python -m unittest discover -s tests -v` 覆盖 API、持久化、前端契约、网关结构和文档输出。
3. 网关压测证据：`output/benchmark/gateway-health.json` 与 `output/benchmark/gateway-chat.json` 保存了实际并发测试结果。

## 7. 项目的创新点如何回答

本文创新点不应夸大为“提出全新算法”，而应定位为工程综合创新：

1. 将 C++ 高并发网络接入与 Python 大模型服务封装结合。
2. 将 Windows 通用演示链路与 Linux/WSL epoll 专项链路明确区分。
3. 将自动化测试、压测原始数据和论文图表统一纳入项目仓库。
4. 通过前端运行状态面板和后台概览降低答辩演示成本。

## 8. 如果老师问系统不足

可以正面回答：

1. 当前大模型能力仍依赖外部 API 或模拟模型，真实推理能力受网络和模型服务影响。
2. C++ 网关当前重点验证 HTTP 转发与并发接入，还不是完整生产级网关。
3. 前端以毕业设计演示为主，复杂权限和运维功能仍可扩展。
4. 独立 Linux 服务器上的更大规模压测可以作为后续工程化优化方向。

## 9. 如果老师问真实模型和扩展能力

当前系统已经预留真实模型接入路径，`.env.example` 默认采用 DeepSeek OpenAI-compatible 配置模板，模型名为 `deepseek-v4-flash`。未配置 API Key 时系统走演示回答；配置 Key 后，Python 服务会按同一 `/api/chat` 契约调用远程模型，失败时写入 `llm_remote_fallback` 日志并回退到演示回答，保证答辩现场主链路不中断。

扩展能力可以这样回答：

1. 多模型配置：通过 `LLM_PROVIDERS_JSON` 维护 provider 清单，后台 `/api/admin/model-providers` 可查看。
2. 知识库增强：`knowledge_base/` 下 Markdown 内容会被检索并作为模型上下文。
3. 流式输出：`/api/chat/stream` 已保留 Server-Sent Events 接口，后续可接前端打字机效果。
4. 并发网关可靠性：C++ 网关使用完整缓冲区发送逻辑，降低短写导致响应不完整的风险。
